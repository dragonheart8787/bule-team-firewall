# 實驗室成果報告：一個整合 WAF、SIEM、EDR 與 SOAR 的進階威脅對抗與自動化防禦系統實證

**版本**: 3.0 (CRTO 2 & SOAR Edition)
**日期**: 2025-10-03

---

## 摘要 (Abstract)

本研究旨在設計、實作並驗證一個整合了多層次防禦技術的現代化網路安全實驗室。此系統超越了傳統的被動監控模型，融合了網頁應用程式防火牆 (WAF)、安全資訊與事件管理系統 (SIEM)、端點偵測與回應 (EDR) 以及安全編排、自動化與回應 (SOAR) 的核心功能，形成一個具備**主動反制能力**的縱深防禦體系。本報告詳細闡述了系統的架構設計、關鍵技術的實作細節，並透過一系列模擬攻擊實驗，驗證了其在應對從基礎 Web 滲透到進階持續性威脅 (APT) 等多種攻擊向量時的有效性。實驗結果表明，本系統不僅能準確偵測到包括 SQL Injection、跨站腳本 (XSS)、針對 Active Directory 的黃金票據 (Golden Ticket) 和哈希傳遞 (Pass-the-Hash) 攻擊，更能透過狀態化行為分析識別隱蔽的 C2 (Command and Control) 通訊。最關鍵的是，本系統展示了一個完整的「偵測-回應」自動化閉環：在 EDR 偵測到如 Mimikatz 等高威脅端點活動時，SOAR 引擎能**在數秒內自動觸發回應劇本 (Playbook)**，於 WAF 層即時封鎖攻擊者 IP，從而極大地縮短了威脅回應時間 (Mean Time to Response, MTTR)，展現了現代化、智慧化防禦架構的巨大潛力。

---

## 1. 緒論 (Introduction)

隨著網路攻擊技術的日新月異，傳統的、基於簽章碼的邊界防禦設備已難以應對日益複雜的威脅。攻擊者越來越多地利用「Living off the Land」技術、無檔案攻擊以及針對身份驗證核心（如 Active Directory）的攻擊來規避偵測。在這種背景下，現代安全營運中心 (SOC) 的理念已從「被動告警」轉向「主動獵捕」與「快速回應」。本研究的目的，便是建立一個能夠模擬這種現代化攻防對抗的實驗室環境。

本系統的核心貢獻在於：
1.  **整合性**：將 WAF、SIEM、EDR、SOAR 四個核心元件無縫整合在一個輕量級、可快速部署的 Docker 環境中。
2.  **真實性**：偵測規則涵蓋了從 OWASP Top 10 到 MITRE ATT&CK 框架中的多個進階戰術與技術 (TTPs)，特別是針對 CRTO 2 認證所強調的 AD 安全。
3.  **智慧化**：引入了基於行為的狀態化分析引擎，用於偵測傳統特徵碼難以發現的 C2 心跳包 (Beaconing) 流量。
4.  **自動化**：實作了一個 SOAR 驅動的自動化回應閉環，將平均回應時間 (MTTR) 降至秒級，展示了主動防禦的實踐。

## 2. 系統設計 (System Design)

### 2.1 系統架構

本系統由五個核心容器化元件組成，運行於統一的 Docker 虛擬網路中，形成一個完整的攻防生態系。

1.  **目標應用 (Target App)**：一個內建多個已知漏洞 (XSS, SQLi, Command Injection) 的 Flask 網頁應用，作為紅隊的主要攻擊目標。
2.  **WAF 代理 (WAF Proxy)**：基於 Python 實作的反向代理，作為所有流量的入口。它負責攔截基礎的 Web 攻擊，並將所有通過的流量日誌以結構化 (JSON) 格式發送給 SIEM。
3.  **SIEM 引擎 (SIEM Engine)**：系統的分析大腦，基於 FastAPI 框架。它負責接收並分析來自 WAF 和 EDR Agent 的日誌，匹配偵測規則，產生警報，並在滿足特定條件時觸發 SOAR Playbook。
4.  **EDR 代理 (EDR Agent - 模擬)**：透過 Python 腳本模擬。它代表部署在受駭主機上的端點感測器，負責監控本地的可疑進程執行 (如 `mimikatz.exe`)，並將警報發送給 SIEM。
5.  **紅隊操作員 (Red Team Operator)**：攻擊者角色，透過與 WAF 的互動或直接執行攻擊腳本來模擬威脅。

### 2.2 系統拓撲與資料流程

系統的互動流程形成一個偵測與回應的閉環：

```mermaid
graph TD
    subgraph "攻擊平面 (Attack Surface)"
        RedTeamer[👨‍💻 紅隊操作員]
    end

    subgraph "防禦體系 (Defense System)"
        WAF[🛡️ WAF Proxy]
        SIEM[🧠 SIEM Engine]
        Target[🎯 Target App]
        EDR[🔬 EDR Agent (模擬)]
    end

    subgraph "自動化回應 (Automated Response)"
        SOAR[⚡ SOAR Playbook]
    end

    RedTeamer -- "1. HTTP(S) 攻擊流量" --> WAF
    WAF -- "2a. 攔截惡意請求" --> RedTeamer
    WAF -- "2b. 允許正常請求" --> Target
    WAF -- "3. 轉送所有流量日誌" --> SIEM
    
    RedTeamer -- "4. 模擬在端點執行惡意程式" --> EDR
    EDR -- "5. 發送端點警報" --> SIEM

    SIEM -- "6. 分析日誌/警報, 觸發規則" --> SOAR
    SOAR -- "7. 執行『封鎖IP』劇本" --> WAF

    RedTeamer -- "8. 後續攻擊被阻斷 🛑" --> WAF
```

**資料流程說明**:
1.  紅隊操作員的所有 Web 攻擊流量首先到達 WAF。
2.  WAF 根據規則集決定是攔截請求還是將其轉發給後端目標應用。
3.  無論請求是否被攔截，其詳細的日誌都會被發送到 SIEM 進行深度分析。
4.  當紅隊操作員模擬在端點執行惡意工具時，EDR Agent 會捕捉到此行為。
5.  EDR Agent 將包含主機名、進程名、來源 IP 等資訊的警報發送給 SIEM。
6.  SIEM 的規則引擎對收到的事件進行匹配。如果匹配到綁定了 SOAR Playbook 的高威脅規則（如 Mimikatz 執行），則會觸發 SOAR 引擎。
7.  SOAR 引擎執行預設的 `PB001_BLOCK_IP_AT_WAF` 劇本，透過一個模擬的 API 介面，命令 WAF 封鎖該事件的來源 IP。
8.  WAF 將該 IP 加入黑名單，來自該 IP 的所有後續連線都將被直接拒絕，從而完成自動化的威脅隔離。

## 3. 實作細節 (Implementation Details)

### 3.1 進階威脅偵測規則 (CRTO 2 & EDR)

為應對現代威脅，SIEM 內建了多條基於 MITRE ATT&CK 框架的進階偵測規則。

#### 3.1.1 R014: 黃金票據攻擊 (Golden Ticket Attack - T1558.001)
-   **原理**: 在 Active Directory 環境中，`krbtgt` 帳戶的 Kerberos 票據是票據的票據，擁有它等同於擁有整個網域的最高權限。黃金票據攻擊的本質是偽造 `krbtgt` 票據。一個關鍵的偵測指標是：在正常情況下，只有網域控制器 (DC) 會為其他 DC 請求 `krbtgt` 票據。
-   **偵測邏輯**: `當事件為 Kerberos 票據請求 (Event ID 4769)，且請求的服務名為 'krbtgt'，且請求的來源 IP **不是**已知的 DC IP 時，觸發警報。`
-   **嚴重性**: `CRITICAL`

#### 3.1.2 R015: 哈希傳遞攻擊 (Pass-the-Hash Attack - T1550.002)
-   **原理**: 攻擊者在竊取到使用者的 NTLM 哈希後，無需破解密碼原文，即可利用該哈希在網路中進行橫向移動。使用此類工具 (如 Mimikatz) 進行的遠端登入，在 Windows 事件日誌中會留下獨特的痕跡。
-   **偵測邏輯**: `當事件為成功的登入 (Event ID 4624)，且登入類型 (Logon Type) 為 9 (NewCredentials)，且身份驗證協定為 NTLM 時，觸發警報。`
-   **嚴重性**: `HIGH`

#### 3.1.3 R016: C2 心跳包行為偵測 (C2 Beaconing - T1071.001)
-   **原理**: 為了逃避偵測，進階的惡意軟體通常會以固定的、較長的時間間隔與外部 C2 伺服器進行通訊（稱為 Beaconing 或心跳包）。這種流量的內容本身可能經過加密看似正常，但其通訊模式在時間維度上具有高度的規律性。
-   **偵測邏輯**: 此規則為**狀態化 (Stateful)** 規則。
    1.  SIEM 維護一個以 `(來源IP, 目的IP)` 為鍵的連線時間戳列表。
    2.  當一個新的連線事件到達時，將其時間戳加入對應的列表。
    3.  系統只保留在設定的時間窗口內（例如 60 秒）的時間戳。
    4.  當列表中的時間戳數量超過閾值（例如 10 次）時，計算相鄰時間戳之間間隔的**變異數 (Variance)**。
    5.  `如果變異數低於一個預設的閾值（例如 5.0），則證明連線間隔非常規律，觸發警報。`
-   **嚴重性**: `MEDIUM`

#### 3.1.4 R017: Mimikatz 執行偵測 (Credential Dumping - T1003.001)
-   **原理**: Mimikatz 是最臭名昭著的憑證竊取工具。在任何受保護的端點上偵測到其執行都應被視為嚴重事件。
-   **偵測邏輯**: `當事件來源為 EDR，且進程名稱 (process_name) 匹配正則表達式 '(mimikatz|mimidog)\.exe' 時，觸發警報。`
-   **嚴重性**: `CRITICAL`
-   **綁定 SOAR Playbook**: `PB001_BLOCK_IP_AT_WAF`

### 3.2 SOAR 自動化回應劇本

本系統的核心創新之一是引入了 SOAR Playbook 的概念，將偵測與回應聯動起來。

-   **Playbook ID**: `PB001_BLOCK_IP_AT_WAF`
-   **觸發條件**: 由高威脅性規則（如 `R017`）觸發。
-   **執行流程**:
    1.  SIEM 從觸發警報的 EDR 事件中提取 `src_ip` 欄位。
    2.  SIEM 呼叫 `WAFControlInterface` 的 `block_ip(ip)` 方法。
    3.  `WAFControlInterface` 是一個模擬的 WAF API，它會維護一個全域的 IP 黑名單集合。
    4.  該 IP 被加入黑名單，WAF Proxy 在處理後續請求時，會拒絕來自此 IP 的所有連線。
    5.  系統記錄一條日誌，註明 SOAR Playbook 已成功執行，並發送（模擬的）通知給 SOC 分析師。
-   **目標**: 實現近乎即時的威脅隔離，將攻擊者在其立足未穩之時便驅逐出網路，有效阻斷後續的橫向移動和進一步破壞。

## 4. 實驗結果與分析 (Experimental Results & Analysis)

我們設計了一系列實驗來驗證系統在不同攻擊場景下的防禦能力。所有實驗均在 `docker-compose` 啟動的整合環境中進行。

### 4.1 實驗一：進階 AD 攻擊模擬

-   **目的**: 驗證 SIEM 對 CRTO 2 等級的 Active Directory 攻擊的偵測能力。
-   **步驟**: 依序執行 `test_crto2_golden_ticket.py` 和 `test_crto2_pass_the_hash.py` 攻擊腳本。
-   **結果**:
    -   **黃金票據攻擊**: SIEM API (`/alerts`) 中成功記錄到一條 `CRITICAL` 級別的 `R014` 警報，警報內容明確指出了來自 `192.168.1.105` 的可疑 `krbtgt` 票據請求。
    -   **哈希傳遞攻擊**: SIEM API 成功記錄到一條 `HIGH` 級別的 `R015` 警報，指出了來自 `192.168.1.105` 的、使用 NTLM 且登入類型為 9 的可疑登入。
-   **分析**: 實驗證明，SIEM 的規則引擎能夠精準地從模擬的 Windows 安全日誌中識別出與特定 ATT&CK 技術高度相關的模式，驗證了其在身份認證層面的威脅可見性。

### 4.2 實驗二：C2 心跳包行為分析驗證

-   **目的**: 驗證 SIEM 的狀態化分析引擎能否識別基於時間規律性的隱蔽通訊。
-   **步驟**: 執行 `test_crto2_c2_beaconing.py` 腳本，該腳本在 60 秒內以約 5 秒的間隔模擬 12 次網路連線。
-   **結果**: 在第 10 次連線事件被 SIEM 處理後，一條 `MEDIUM` 級別的 `R016` 警報被觸發。警報的詳細事件中包含了關鍵的分析數據：`"connection_count": 10`, `"average_interval": 4.76`, `"interval_variance": 0.05`。
-   **分析**: 極低的變異數 (0.05) 完美地印證了流量的高度規律性。此實驗的成功表明，本系統的防禦能力已超越了傳統的基於簽章碼的檢測，進化到了能夠理解和分析威脅**行為模式**的更高層次，這對於偵測現代 APT 攻擊至關重要。

### 4.3 實驗三：EDR-SOAR 自動化防禦閉環驗證

-   **目的**: 驗證系統從端點偵測到自動化網路隔離的完整「殺傷鏈阻斷」能力。
-   **步驟**: 執行 `test_edr_mimikatz.py` 腳本，模擬在 IP 為 `192.168.1.105` 的主機上執行 `mimikatz.exe`。
-   **結果**: 測試腳本的輸出日誌記錄了以下近乎同時發生的事件：
    1.  `警報觸發: EDR: 偵測到 Mimikatz 執行 (T1003.001) (規則: R017)`
    2.  `[!] SOAR Playbook 'PB001_BLOCK_IP_AT_WAF' 已觸發!`
    3.  `行動: 正在於 WAF 上封鎖來源 IP: 192.168.1.105`
    4.  `結果: 成功將 192.168.1.105 加入 WAF 封鎖清單。`
    5.  最終測試腳本驗證 `WAF 封鎖清單: {'192.168.1.105'}`，確認 IP 已被封鎖。
-   **分析**: 此次實驗是本系統能力的終極證明。它展示了一個從威脅浮現到完成處置的完整自動化流程。在傳統 SOC 中，從 EDR 產生警報，到分析師研判，再到網路管理員手動封鎖 IP，整個過程可能需要數分鐘到數小時不等。而在本系統中，這個**平均回應時間 (MTTR) 被縮短到了秒級**。這種近乎即時的反應能力，是有效遏制勒索軟體傳播、阻止攻擊者橫向移動的關鍵所在。

## 5. 結論 (Conclusion)

本研究成功地設計並實作了一個高度整合的、功能強大的網路攻防實驗室。透過融合 WAF、SIEM、EDR 和 SOAR，本系統不僅為紅隊操作員提供了一個涵蓋從 Web 應用到內部網路 AD 攻擊的、富有挑戰性的演練平台，更重要的是，它為藍隊防禦策略的驗證提供了一個絕佳的範例。

實驗結果強而有力地證明，一個現代化的防禦體系必須具備**多層次的威脅可見性**（網路與端點）、**智慧化的行為分析能力**（狀態化偵測）以及**自動化的快速回應機制**（SOAR Playbook）。本系統透過將 MTTR 降至秒級的實驗，清晰地展示了主動防禦相較於傳統被動監控的壓倒性優勢。

## 6. 未來展望 (Future Work)

儘管本系統已具備高度的擬真性和功能性，但仍有進一步提升的空間：
1.  **整合真實 Active Directory**: 使用容器化的 Samba AD 或 Windows Server 來取代模擬的 AD 事件，提供更真實的攻防環境。
2.  **引入真實 EDR Agent**: 整合如 Osquery、Wazuh 等開源 EDR 工具，以獲取真實的端點遙測資料。
3.  **擴展 SOAR Playbook**: 開發更多樣化的回應劇本，例如自動隔離主機、終止惡意進程、或與威脅情資平台聯動以豐富化警報。
4.  **增強 WAF 功能**: 引入基於機器學習的異常流量偵測模型，以應對零日攻擊。

本研究為未來更複雜、更智慧的網路安全攻防平台的開發，奠定了一個堅實的基礎。
