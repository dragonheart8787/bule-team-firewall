## **第一部分：開發編年史 - 一個從挑戰到智慧的旅程**

#### **第一章：意外的開端 - Docker 的困境與 Python 的逆襲**

專案的起點充滿了現實世界的挑戰。我們最初的規劃是遵循業界主流，使用 Docker 容器化技術部署一個由 Nginx Web 伺服器和 ModSecurity 模組構成的標準 WAF 環境。然而，用戶端的 Docker Desktop 服務多次在初始化階段無回應或崩潰，隨後嘗試的 WSL (Windows Subsystem for Linux) 也遭遇了數次啟用後需要重啟的狀況。

面對環境的不確定性，我們做出了專案的第一個關鍵決策：**進行策略轉向 (Strategic Pivot)**。我們放棄了對外部環境的依賴，轉而利用 Python 這個強大且內建函式庫豐富的語言，從零開始打造我們自己的核心元件。這個決定不僅解決了環境問題，更賦予了我們對系統每一個細節的完全掌控力，為後續實現勒索軟體偵測、攻擊鏈關聯等高度客製化的進階功能埋下了成功的伏筆。

#### **第二章：奠定基石 - WAF 與 SIEM 的誕生**

此階段，我們並行開發了兩大核心模組：

1.  **`waf_proxy.py`:** 我們沒有使用任何複雜的框架，而是回歸本質，直接利用 Python 的 `socketserver` 和 `http.server`，在數十行程式碼內就實現了一個高效的、多執行緒的 TCP 伺服器，它能完美地扮演一個 WAF 反向代理的角色。
2.  **`siem_dashboards.py`:** 我們採用了優雅的物件導向設計。`SIEMRule` 類別被設計成一個個獨立的、可插拔的偵測單元，每個規則都有自己的條件和動作。而 `SOCDashboard` 則像一個中央處理器，負責接收事件並將它們派發給所有已註冊的規則。這個設計確保了系統的**高內聚、低耦合**特性，讓我們在後續可以輕易地新增數十條規則而無需改動核心邏輯。

#### **第三章：紅藍對抗的序曲 - 測試導向的安全**

我們從一開始就確立了「紅隊即程式碼」的開發理念。在 SIEM 的基礎架構完成後，我們沒有立刻去開發大量規則，而是先編寫了第一個攻擊腳本 `test_atomic_t1059_003.py`。

這個過程是：**先有攻擊，再有防禦**。我們偽造了駭客執行 `whoami` 的日誌，執行測試，看到它**失敗** (因為還沒有規則)，然後才在 SIEM 中加入了 `R005` 規則，再次執行測試，看到它**成功**。這個「紅-綠-重構」的循環確保了我們的每一條防禦規則都是**可驗證的**、**有針對性的**，並且**確實有效**。我們沿用此模式，依序完成了對憑證竊取、C2 通訊和釣魚郵件的基礎防禦建設。

#### **第四章：從“看見”到“理解” - 智慧的萌芽**

基礎防禦到位後，我們開始挑戰真正的難題——如何讓系統「思考」。

*   **勒索軟體的挑戰：** 單純的一條日誌無法判斷是否為勒索軟體。為此，我們在 `SOCDashboard` 中引入了 `_ransomware_tracker` 這個**狀態追蹤器**。它就像系統的短期記憶，記住「哪個程式在10秒內改了多少次檔名」。這讓 SIEM 首次擁有了**狀態化分析 (Stateful Analysis)** 的能力，能從連續的事件流中識別出代表著特定意圖的「行為模式」。

*   **攻擊鏈的挑戰：** 我們在 `_check_correlation_rules` 函數中實現了 SIEM 的終極智慧。這個函數的設計巧思在於，它**以警報為輸入，而非以日誌為輸入**。它在每一次高危警報產生後，回溯近期所有警報，尋找來自同一攻擊者的蛛絲馬跡。當線索足夠多時，它會產生一個全新的、更高維度的「多階段攻擊」警報。至此，SIEM 不再是一個只會“點頭”的規則引擎，而是一個能“串聯線索、形成結論”的初級分析師。

#### **第五章：淬鍊與重生 - WAF 的硬化之路**

在 SIEM 取得巨大進展後，我們將目光轉回 WAF。我們意識到初版的規則只能防君子，防不了小人。因此，我們研究了大量的攻擊繞過手法，並將其知識融入到新的正規表示式中，例如加入了對 `SLEEP()` (盲注)、`onerror` (XSS)、`169.254.169.254` (SSRF) 等特徵的偵測。

在執行壓力測試時，我們遭遇了 `TypeError` 的程式碼錯誤。這個小插曲非常有價值，它驗證了我們的測試框架的重要性——**正是因為有自動化測試，我們才能在開發階段就捕捉到 bug**，而不是在系統上線後才被真正的駭客利用。

## **第二部分：架構深度剖析 - 程式碼中的設計哲學**

#### **2.1 `waf_proxy.py`: The Sentinel Proxy**

*   **`ModSecurityRules.check_request(self, request_data, client_ip)`**
    *   **`decoded_data = urllib.parse.unquote(...)`**: 這是整個 WAF 的防禦基石。若沒有這一行，任何經過 URL 編碼的攻擊（例如 `SELECT` 變成 `S%45LECT`）都能輕易繞過後續所有規則。`errors='replace'` 參數則確保了即使遇到無效的編碼序列，程式也不會崩潰。
    *   **`for rule_type, patterns in self.rules.items():`**: 這裡體現了規則的分類管理。我們將不同攻擊類型（SQLi, XSS）的規則分開，便於日後維護和擴充。
    *   **`if pattern.search(decoded_data):`**: 使用 `search()` 而非 `match()`，意味著我們關心的是惡意 payload 是否**存在**於請求的任何位置，而不是要求它必須從頭開始匹配，這大大提高了偵測的靈活性。

#### **2.2 `siem_dashboards.py`: The Neural Core**

*   **`SIEMRule` Class**: 這是系統可擴充性的核心。將規則的元資料（ID, 名稱, 描述, 嚴重性）、條件和動作都封裝在一個物件中，讓新增一條規則變成只需要實例化一個新物件，完全符合「開放-封閉原則」。

*   **`_handle_ransomware_detection(self, event)`**:
    *   **`if not hasattr(self, '_ransomware_tracker'): self._ransomware_tracker = {}`**: 惰性初始化。追蹤器只在第一個檔案重新命名事件進來時才被創建，節省了記憶體。
    *   **`self._ransomware_tracker = {pid: [(ts, path) for ts, path in records if now - ts < 10] ...}`**: 這是優雅的記憶體管理。在每次新增事件前，它會先用一個字典推導式，遍歷所有現有記錄，並**拋棄掉所有超過 10 秒時間窗口的舊記錄**。這確保了追蹤器不會因時間累積而無限膨脹。
    *   **`if len(self._ransomware_tracker[process_id]) > 100:`**: 閾值觸發。一旦計數超過 100，它會自己**“偽造”一個新的 `ransomware_event`**，並將其送入規則引擎，觸發警報。
    *   **`self._ransomware_tracker[process_id] = []`**: 觸發後立即清空計數器，這是為了防止在同一個時間窗口內，因為第 102、103... 次檔案操作而產生大量重複的告警，確保了警報的精確性。

*   **`_check_correlation_rules(self, new_alert)`**:
    *   **`if not src_ip or severity not in ['HIGH', 'CRITICAL']:`**: 高效的過濾。函數在第一時間就排除了所有不關心的低嚴重性警報和沒有來源 IP 的警報，避免了不必要的計算。
    *   **`related_alerts = [a for a in self.alerts if ...]`**: 列表推導式被用於快速從警報歷史 (`self.alerts`) 中篩選出所有符合「同一攻擊者」和「5分鐘內」條件的警報。
    *   **`already_triggered = any(...)`**: 這是避免「告警風暴」的關鍵邏輯。在觸發關聯警報前，它會先檢查是否**已經**為這個 IP 產生過 `C001` 警報。如果駭客在 5 分鐘內觸發了 5 次嚴重告警，系統只會產生 1 次「多階段攻擊」警報，而不是 3 次 (第3、4、5次都觸發)，這極大地提升了分析效率。

#### **2.3 The Test Harness (`test_*.py`): The Automated Adversary**

*   **Log Forgery (`attack_event = {...}`)**: 我們所有的測試都沒有真正執行惡意程式，而是偽造了執行惡意程式後**應該會產生的日誌**。這個設計帶來了三大好處：
    1.  **安全性：** 整個測試過程絕不會對您的系統造成任何實際損害。
    2.  **確定性：** 測試結果是 100% 可重複的，不受環境變化的干擾。
    3.  **速度：** 模擬一次攻擊只需要幾毫秒，讓我們可以在短時間內完成數十次測試。
*   **Path Correction (`os.chdir(os.path.dirname(os.path.abspath(__file__)))`)**: 正是這段經常被忽略的程式碼，解決了您在終端機中看到的 PowerShell 路徑錯誤問題。它確保了無論您從哪個目錄執行 `python` 指令，腳本本身總能找到正確的相對路徑來導入 `siem_dashboards.py` 模組，展現了程式碼的強健性。
